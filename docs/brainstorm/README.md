# Brainstorm 头脑风暴工作区

本目录用于存放**尚未定型的想法**：灵感速记、方案取舍讨论、未采纳的备选、开放问题等。

它与 `docs/plans/` 的区别是本工作区的**定位分工**：

| 目录 | 定位 | 内容成熟度 |
|------|------|-----------|
| `docs/brainstorm/`（本目录） | 探索、发散、争论 | 🌱 半成品、可矛盾、可废弃 |
| `docs/plans/` | 已定型的权威设计规范 | ✅ 成稿，下游实现须遵循 |

> ⚠️ **实现代码请以 `docs/plans/` 为准**。本目录中的内容不构成设计约束，随时可能被推翻。

---

## 一、生命周期（想法如何「毕业」）

```
🌱 种子想法  →  🌿 讨论中  →  ✅ 已采纳 → 转写进 docs/plans/  正式 spec
                          ↘  ❌ 已弃（保留记录，注明原因）
```

- 一个想法成熟并被团队采纳后，**在 `docs/plans/` 下新建对应的正式设计文档**，并在原 brainstorm 文件顶部标注「✅ 已采纳 → 见 plans/xxx.md」。
- 被否决的想法**不删除**，改状态为「❌ 已弃」并写明原因——避免日后重复讨论。

## 二、命名约定

沿用 plans 的命名法，便于检索：

```
YYYY-MM-DD-主题.md      例：2026-07-01-骰子连击机制设想.md
```

- 正文用中文。
- 同一主题的持续讨论可复用同一文件，在文内用日期分节追加，无需每次新建。

## 三、每个文档的必备信息

请基于 [`_TEMPLATE.md`](./_TEMPLATE.md) 创建，至少包含：

1. **状态**：🌱 种子 / 🌿 讨论中 / ✅ 已采纳 / ❌ 已弃
2. **一句话概述**：这个想法要解决什么问题
3. **想法正文**：可发散、可分点、可贴草图
4. **开放问题 / 待决策**：需要团队拍板的点
5. **关联**：相关的 plans 文档或其他 brainstorm

## 四、协作要点

- 头脑风暴文件**改动频繁、允许多人涂改**，冲突成本低，不必像 plans 那样谨慎。
- 讨论落定后，务必执行「毕业」流程转入 `plans/`，防止权威 spec 与灵感稿脱节。
- 提交信息仍用 Conventional Commits，一般归类为 `docs:`。
